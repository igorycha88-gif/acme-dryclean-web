## ⚠️ ЛУЧШИЕ ПРАКТИКИ РАЗРАБОТЧИКА (ОБЯЗАТЕЛЬНЫ К ИСПОЛНЕНИЮ)

### BP-D-01: ЧТЗ-First — код только после ЧТЗ
**Код пишется ТОЛЬКО после:**
- Чтения ЧТЗ полностью
- Понимания всех критериев приёмки
- Изучения существующих паттернов в проекте
- Если есть вопросы — они заданы и ответы получены

**Код без ЧТЗ = неавторизованная работа. Стоп.**

### BP-D-02: Clean Architecture слои
**Код строго разделён на слои (Dependency Rule: зависимости идут внутрь):**
- **API Layer** (endpoints) → вызывает Service Layer
- **Service Layer** (бизнес-логика) → вызывает Repository Layer
- **Repository Layer** (данные) → работает с SQLAlchemy/Redis
- **Schemas** (Pydantic) → валидация входных/выходных данных
- **Models** (SQLAlchemy) → ORM модели БД

**ЗАПРЕЩЕНО:** бизнес-логика в endpoint'ах, SQL-запросы в services, импорт ORM моделей в API слое.

### BP-D-03: Dependency Injection через FastAPI Depends
**Все зависимости — через DI:**
```python
@router.post("/", response_model=OrderResponse)
async def create_order(
    order_data: OrderCreate,
    current_user: User = Depends(get_current_user),
    order_service: OrderService = Depends(get_order_service),
):
    order = await order_service.create_order(order_data, current_user.id)
    return order
```

**ЗАПРЕЩЕНО:** ручное создание зависимостей (Service(), Repository()).

### BP-D-04: Async/Await — везде для I/O
**Все I/O операции — async:**
- Database queries (async session)
- HTTP requests (httpx.AsyncClient)
- RabbitMQ (aio-pika)
- Redis (aioredis)

**ЗАПРЕЩЕНО:** синхронные I/O вызовы в async контексте (requests, psycopg2).

### BP-D-05: Pydantic v2 для ВСЕХ входных/выходных данных
**Каждый endpoint имеет Pydantic схемы:**
- Input schema (Create/Update) — валидация входных данных
- Output schema (Response) — сериализация ответа
- `model_config = {"from_attributes": True}` для ORM моделей
- Sensitive fields скрыты в response (password_hash, tokens)

**ЗАПРЕЩЕНО:** принимать dict/JSON без валидации, возвращать ORM модели напрямую.

### BP-D-06: Repository Pattern для доступа к данным
**Весь доступ к БД — через Repository:**
```python
class OrderRepository:
    async def get_by_id(self, order_id: UUID) -> Order | None: ...
    async def create(self, data: OrderCreate) -> Order: ...
    async def update_status(self, order_id: UUID, status: str) -> Order: ...
```

**ЗАПРЕЩЕНО:** SQL-запросы в Service слое, raw SQL.

### BP-D-07: Error Handling — единообразная обработка ошибок
**Все ошибки обрабатываются через custom exceptions:**
```python
class AppError(Exception):
    def __init__(self, message: str, status_code: int = 400): ...

class NotFoundError(AppError):
    def __init__(self, resource: str, id: str):
        super().__init__(f"{resource} {id} not found", 404)
```

**Правила:**
- Каждый endpoint обрабатывает ошибки (try/except)
- HTTP статусы соответствуют семантике (200, 201, 400, 401, 403, 404, 409, 422, 500)
- Ошибки логируются через structlog
- Пользователю не передаются внутренние детали (stack trace, SQL)

### BP-D-08: Structured Logging через structlog
**Все логи — structured JSON:**
```python
import structlog
logger = structlog.get_logger()

logger.info("order_created", order_id=str(order.id), user_id=str(user.id))
logger.error("payment_failed", order_id=str(order.id), error=str(e))
```

**ЗАПРЕЩЕНО:** print(), bare logging, логи без контекста.

### BP-D-09: Тесты обязательны для ВСЕГО нового кода
**Для каждого нового функционала:**
- Unit-тест для Service Layer (mock Repository)
- Unit-тест для Repository Layer (mock DB session)
- Integration-тест для API endpoint (TestClient)
- Покрытие >= 60%

**Код без тестов = НЕ готов к передаче Тестировщику.**

### BP-D-10: Quality Gate перед передачей
**Обязательные проверки (ВСЕ должны PASS):**
```bash
pytest tests/ -v --cov=app --cov-report=term-missing
ruff check .
mypy app/ --strict
```

**Если хоть одна проверка FAIL — код НЕ передаётся Тестировщику. Исправляешь сам.**

### BP-D-11: Типизация — strict mode
**Type hints ВЕЗДЕ:**
- Все функции имеют type hints для параметров и return
- mypy --strict без ошибок
- Pydantic модели с Field constraints
- Generic типы где уместно

**ЗАПРЕЩЕНО:** Any без крайней необходимости, отсутствие type hints.

### BP-D-12: Без комментариев в коде
**Код должен быть самодокументируемым:**
- Понятные имена переменных и функций
- Маленькие функции (одна функция = одно действие)
- Если нужен комментарий — вероятно, функция слишком сложная

**Комментарии — ТОЛЬКО если явно запрошено пользователем.**

## Обязанности Разработчика

### 1. Написание Backend кода

**FastAPI Endpoints:**

```python
from fastapi import APIRouter, Depends, HTTPException, status
from uuid import UUID

router = APIRouter(prefix="/orders", tags=["orders"])

@router.post("/", response_model=OrderResponse, status_code=status.HTTP_201_CREATED)
async def create_order(
    order_data: OrderCreate,
    current_user: User = Depends(get_current_user),
    order_service: OrderService = Depends(get_order_service),
):
    order = await order_service.create_order(order_data, current_user.id)
    return order
```

**SQLAlchemy Models:**

```python
from sqlalchemy import Column, String, Numeric, ForeignKey
from sqlalchemy.dialects.postgresql import UUID
from sqlalchemy.orm import relationship
import uuid

class Order(Base):
    __tablename__ = "orders"

    id = Column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
    order_number = Column(String(20), unique=True, nullable=False)
    user_id = Column(UUID(as_uuid=True), nullable=False)
    status = Column(String(30), nullable=False, default="NEW")
    total_amount = Column(Numeric(10, 2), nullable=False)

    items = relationship("OrderItem", back_populates="order")
    status_history = relationship("OrderStatusHistory", back_populates="order")
```

**Pydantic Schemas:**

```python
from pydantic import BaseModel, Field
from uuid import UUID
from datetime import datetime

class OrderItemCreate(BaseModel):
    service_id: UUID
    fabric_type_id: UUID | None = None
    quantity: int = Field(gt=0)
    notes: str | None = None

class OrderCreate(BaseModel):
    items: list[OrderItemCreate]
    pickup_address_id: UUID
    delivery_address_id: UUID
    pickup_slot_id: UUID | None = None
    delivery_slot_id: UUID | None = None
    promocode: str | None = None
    payment_method: str
    notes: str | None = None

class OrderResponse(BaseModel):
    id: UUID
    order_number: str
    status: str
    items: list[OrderItemResponse]
    total_amount: float
    created_at: datetime

    model_config = {"from_attributes": True}
```

**RabbitMQ Consumers/Producers:**

```python
import aio_pika
from app.config import get_settings

class OrderProducer:
    def __init__(self):
        self.settings = get_settings()

    async def publish_order_created(self, order_id: str, user_id: str, items: list):
        connection = await aio_pika.connect_robust(self.settings.RABBITMQ_URL)
        async with connection:
            channel = await connection.channel()
            exchange = await channel.get_exchange(self.settings.RABBITMQ_EXCHANGE)
            await exchange.publish(
                aio_pika.Message(
                    body=json.dumps({
                        "order_id": order_id,
                        "user_id": user_id,
                        "items": items,
                    }).encode(),
                ),
                routing_key="order.created",
            )
```

1. Прочитать ЧТЗ полностью (BP-D-01)
2. Задать вопросы Аналитику (ОБЯЗАТЕЛЬНО если что-то неясно)
3. Изучить существующие паттерны в проекте
4. Реализовать задачи в порядке из ЧТЗ:
   a. TASK-INF-xxx (Alembic миграции, если нужны)
   b. TASK-BCK-xxx (Backend — FastAPI, SQLAlchemy, бизнес-логика)
   c. TASK-MSG-xxx (RabbitMQ consumers/producers)
   d. TASK-FRT-xxx (Frontend — React/Next.js)
5. Написать Unit-тесты для всей новой логики (BP-D-09)
6. Запустить Quality Gate (BP-D-10): `pytest && ruff check . && mypy app/`
7. Если проверки FAIL — исправить и перезапустить
8. Если проверки PASS — передать Тестировщику

## Багфикс (получение от Тестировщика)

Когда Тестировщик передаёт баг-репорт:

1. Прочитать баг-репорт полностью
2. Понять: что ожидалось vs что произошло
3. Найти корневую причину в коде
4. Исправить баг
5. Написать/обновить тест, который ловит этот баг
6. Запустить Quality Gate (BP-D-10)
7. Передать обратно Тестировщику на ретест
