# ЧТЗ: Сохранность аналитических данных при деплое

## Бизнес-контекст

После деплоя на проде Grafana показывает изменение метрики «Посетители» с 2 до 1.
Пользователь считает, что данные перезатираются при деплое.

## Реальная ситуация

**Данные в БД НЕ перезатираются.** PostgreSQL volume (`dryclean_postgres`) сохраняется.
Причина изменения метрики — комбинация двух факторов:
1. Grafana использует скользящее окно `NOW() - INTERVAL '24 hours'` — события стареют и выпадают
2. Tracking service недоступен во время деплоя — события с сайта теряются бесследно

## Задачи

### TASK-FRT-001: Буферизация событий в трекере (frontend)

**Файл:** `frontend/src/lib/tracker.ts`

**Проблема:** `fetch().catch(() => {})` — события теряются при недоступности tracking service.

**Решение:** Реализовать очередь событий с повторными попытками:
- При ошибке отправки — сохранять события в `localStorage` (ключ `da_tracker_queue`)
- При успешной отправке — очищать очередь
- Максимум 50 событий в очереди (FIFO, oldest dropped)
- Повторная отправка раз в 5 секунд
- Отправка батчом (массив событий) при наличии очереди

**Критерии приёмки:**
- [ ] При недоступности сервера события сохраняются в localStorage
- [ ] При восстановлении связи очередь отправляется
- [ ] Очередь ограничена 50 событиями
- [ ] Нет memory leak (очистка после отправки)

### TASK-FRT-002: Batch API endpoint для событий (backend)

**Файл:** `backend/services/tracking/app/api/v1/endpoints/track.py`

**Проблема:** Есть только `/api/v1/tracking/event` для одиночных событий. Для батча нужен отдельный endpoint.

**Решение:** Добавить `POST /api/v1/tracking/events/batch`:
- Принимает массив событий (макс 50)
- Сохраняет все события транзакционно
- Возвращает `{ "processed": N }`

**Критерии приёмки:**
- [ ] Endpoint обрабатывает до 50 событий за запрос
- [ ] Транзакционность: все или ничего
- [ ] Валидация каждого события через существующую Pydantic схему
- [ ] Unit-тесты

### TASK-BCK-001: Shared PostgreSQL для blue-green (инфраструктура)

**Файл:** `docker-compose.blue.yml`, `docker-compose.green.yml`

**Проблема:** Blue и Green используют раздельные PostgreSQL volumes — аналитика не переносится при переключении.

**Решение:** Вынести PostgreSQL в отдельный shared сервис (docker-compose.monitoring.yml уже существует как шаблон):
- Создать `docker-compose.shared-db.yml` с единым PostgreSQL
- Blue и Green compose файлы ссылаются на shared PostgreSQL через external network
- Альтернативно: в `deploy.sh` добавлен шаг миграции данных analytics_* таблиц между blue/green БД

**Критерии приёмки:**
- [ ] При переключении blue↔green аналитические данные сохраняются
- [ ] Content-данные переносятся корректно

### TASK-INF-001: Улучшение Grafana дашборда — кумулятивные метрики

**Файл:** `monitoring/grafana/dashboards/business_analytics.json`, `grafana/dashboards/business_analytics.json`

**Проблема:** Все панели используют только скользящее окно 24ч, нет понимания «сколько всего было».

**Решение:** Добавить панели:
- «Всего посетителей» — `SELECT COUNT(DISTINCT session_id) FROM analytics_events` (без WHERE, всё время)
- «Всего событий» — `SELECT COUNT(*) FROM analytics_events`
- Обновить описание панели «Посетители» — явно указать «за последние 24 часа (скользящее окно)»

**Критерии приёмки:**
- [ ] Добавлены панели с кумулятивными метриками
- [ ] Описание панелей явно указывает период (24ч / всего)
