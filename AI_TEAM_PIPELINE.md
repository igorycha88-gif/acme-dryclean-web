## 3. Гарантия непрерывности конвейера

### Принцип: от идеи до развёрнутой задачи

**Конвейер обеспечивает ПОЛНЫЙ цикл:**
```
Идея (бизнес-задача)
  → Анализ (Аналитик)
  → Проектирование (Архитектор, если нужно)
  → ЧТЗ + декомпозиция (Аналитик)
  → Реализация (Разработчик)
  → Проверка (Тестировщик)
  → [цикл багфикса если нужно]
  → Пересборка + деплой (DevOps)
  → Развёрнутая задача (локально)
```

**Переходы между этапами — АВТОМАТИЧЕСКИЕ. Без пауз. Без ручного подтверждения.**

**Конвейер НЕ останавливается пока задача не развёрнута локально.**

## 4. Золотое правило: ОБЯЗАТЕЛЬНЫЕ ВОПРОСЫ

**Каждая роль ОБЯЗАНА задать вопросы перед началом работы. Без ответов работа НЕ начинается.**

### 4.1 Почему это критично

- Предотвращает реализацию не того, что нужно
- Выявляет скрытые требования на ранних этапах
- Обеспечивает взаимопонимание между ролями
- Снижает количество переделок

### 4.2 Протокол согласования

```
┌──────────┐    Вопросы     ┌──────────┐    Ответы     ┌──────────┐
│   Роль   │ ────────────── │ Команда  │ ───────────── │   Роль   │
│(инициатор)│                │(адресат) │               │(инициатор)│
└──────────┘                └──────────┘               └──────────┘
                                                            │
                                                      Подтверждение
                                                      "Могу начать"
                                                            │
                                                            ▼
                                                      НАЧАЛО РАБОТЫ
```

### 4.3 Формат запроса на согласование

```markdown
## Запрос на согласование: [Роль] → [Этап]

**Задача:** [Описание]
**ЧТЗ:** [Ссылка]

### Мои вопросы:
1. [Вопрос 1]
2. [Вопрос 2]
3. [Вопрос 3]

### Мой план:
- [ ] Шаг 1
- [ ] Шаг 2
- [ ] Шаг 3

**Статус:** Жду подтверждения.
```

**РАБОТА НАЧИНАЕТСЯ ТОЛЬКО ПОСЛЕ ПОЛУЧЕНИЯ ПОДТВЕРЖДЕНИЯ.**

```
1. Прочитать ЧТЗ полностью (BP-D-01)

### ЭТАП 2.5: Цикл багфикса (Тестировщик → Разработчик → Тестировщик)

**При обнаружении багов (NO-GO) запускается цикл:**

```
Тестировщик нашёл баги (NO-GO)
         │
         ▼
Регистрирует BUG-XXX с severity/priority
         │
         ▼
Передаёт Разработчику с полным баг-репортом
         │
         ▼
Разработчик:
  1. Читает баг-репорт
  2. Находит корневую причину
  3. Исправляет баг
  4. Пишет/обновляет тест
  5. Quality Gate: pytest + ruff + mypy
         │
         ▼
Передаёт обратно Тестировщику
         │
         ▼
Тестировщик (ретест):
  1. Баг исправлен? (Verification)
  2. Ничего не сломано? (Regression)
  3. Межсервисные контракты целы? (Integration)
         │
     ┌───┴───┐
     │       │
   PASS    FAIL
     │       │
     ▼       ▼
   Закрыть  Переоткрыть
    баг     баг → Разработчику
     │
     ▼ (все баги закрыты)
   GO → DevOps
```

**Цикл повторяется пока ВСЕ баги не закрыты. GO с открытыми Critical/Blocker — ЗАПРЕЩЁН.**

**Максимальное количество итераций: 10. Если более 3 итераций — эскалация к Аналитику.**

## 6. Разбиение больших задач на этапы

### 6.1 Критерий "большой задачи"

Задача считается **большой**, если:
- Декомпозиция содержит **более 10 задач** (TASK-XXX)
- Оценка времени **более 16 часов**
- Затронуто **более 3 сервисов** (микросервисная специфика)
- Требуется **изменение архитектуры**
- Затронуты **межсервисные контракты** (RabbitMQ events)

### 6.2 Принцип разбиения

Большая задача разбивается на **подэтапы**, каждый из которых проходит полный цикл конвейера:

```
БОЛЬШАЯ ЗАДАЧА
├── Подэтап 1 (Фундамент — БД + API контракты)
│   ├── Аналитик → ЧТЗ-1

## 7. Артефакты конвейера

### 7.1 Жизненный цикл задачи

```
BUSINESS_TASK_FORM.md          ← Вход: бизнес-задача
        │
        ▼
ЧТЗ документ                    ← Аналитик: требования (BP-A-01 — BP-A-10)
        │
        ▼
ADR документ                    ← Архитектор: решение (если нужно) (BP-AR-01 — BP-AR-10)
        │
        ▼
TASK-XXX-YYY список             ← Аналитик: декомпозиция
        │
        ▼

        │
        ▼
Test Summary Report             ← Тестировщик: проверка (BP-T-01 — BP-T-10)
        │
        ▼
Deployment Report               ← DevOps: пересборка + итоги (BP-OPS-01 — BP-OPS-10)
        │
        ▼
Развёрнутая задача локально     ← РЕЗУЛЬТАТ
```

### 7.2 Хранение артефактов

| Артефакт | Где хранится | Кто создаёт |
|----------|-------------|-------------|
| Бизнес-задача | `BUSINESS_TASK_FORM.md` | Заказчик |

| ADR | `ADR-[ID]-[Название].md` | Архитектор |
| Декомпозиция | Внутри ЧТЗ, раздел 7 | Аналитик |

| Тесты | `services/[service-name]/tests/` | Разработчик + Тестировщик |

| Deployment Report | Выводится в консоль | DevOps |

## 8. Правила конвейера (ОБЯЗАТЕЛЬНЫ К СОБЛЮДЕНИЮ)

### 8.1 Правило порядка

> **Аналитик — ПЕРВЫЙ. Архитектор — по решению Аналитика. DevOps — ПОСЛЕДНИЙ.**

### 8.2 Правило вопросов

> **Каждая роль ДОЛЖНА задать минимум 3 вопроса перед началом работы.**
>
> Без ответов — работа НЕ начинается.

### 8.3 Правило ЧТЗ

> **Разработчик НЕ пишет код без утверждённого ЧТЗ.**
>
> Если ЧТЗ неоднозначен — Разработчик возвращает Аналитику на доработку.

### 8.4 Правило тестов

> **Разработчик НЕ передаёт работу Тестировщику без прохождения `pytest && ruff check . && mypy app/`.**
>
> Если тесты падают — работа возвращается Разработчику.

### 8.5 Правило приёмки

> **Тестировщик НЕ ставит GO без проверки ВСЕХ критериев приёмки из ЧТЗ.**
>
> Каждый критерий приёмки — минимум один тест-кейс.

### 8.6 Правило регрессии

> **После каждого фикса Тестировщик проверяет не только баг, но и окружающий функционал + межсервисные контракты.**

### 8.7 Правило цикла багфикса

> **Баги → Разработчик → фикс → Quality Gate → Тестировщик → ретест. Цикл повторяется до GO.**
>
> **GO с открытыми Critical/Blocker багами — ЗАПРЕЩЁН.**

### 8.8 Правило полной пересборки

> **DevOps ВСЕГДА делает ПОЛНУЮ пересборку ВСЕХ сервисов. ЗАПРЕЩЕНА частичная пересборка.**

### 8.9 Правило подведения итогов

> **DevOps ВСЕГДА выводит итоговый отчёт после деплоя: что пересобрано, что развёрнуто, статус всех сервисов.**

### 8.10 Правило непрерывности

> **Конвейер НЕ останавливается пока задача не развёрнута локально.**
>
> **Переходы между этапами — АВТОМАТИЧЕСКИЕ.**

### 8.11 Правило документации

> **Разработчик обновляет документацию (ARCHITECTURE.md, API_SPECIFICATION.md) при изменении архитектуры.**

### 8.12 Правило лучших практик

> **Лучшие практики (BP-*) из каждого скилла ОБЯЗАТЕЛЬНЫ к исполнению.**
>
> **Нарушение best practice = возврат на доработку.**

## 10. Быстрый старт (для Заказчика)

### Чтобы запустить конвейер:

1. **Заполните форму:** `BUSINESS_TASK_FORM.md` — опишите бизнес-задачу
2. **Передайте Аналитику:** Аналитик начнёт с вопросов к вам
3. **Отвечайте на вопросы:** Каждая роль будет задавать вопросы — отвечайте
4. **Утвердите ЧТЗ:** Проверьте требования перед разработкой
5. **Получите результат:** После прохождения всех этапов — развёрнутая задача локально

### Что ожидать:

| Этап | Срок | Ваше участие |

## 11. Специфика микросервисной архитектуры DryClean Pro

### 11.1 Межсервисные контракты

При изменении контрактов между сервисами (RabbitMQ events, API calls):

1. **Аналитик** явно указывает затронутые сервисы
2. **Архитектор** проектирует backward-compatible изменения
3. **Разработчик** реализует изменения во всех затронутых сервисах
4. **Тестировщик** проверяет интеграцию между сервисами
5. **DevOps** деплоит ВСЕ затронутые сервисы одновременно (полная пересборка)

### 11.2 Миграции БД

Каждый сервис имеет свою БД и свои Alembic миграции:

```
services/
├── user-service/alembic/
├── auth-service/alembic/
├── order-service/alembic/
├── catalog-service/alembic/
├── payment-service/alembic/
├── notification-service/alembic/
├── delivery-service/alembic/
├── loyalty-service/alembic/
├── review-service/alembic/
└── pricing-service/alembic/
```

### 11.3 Синхронный vs Асинхронный вызов

| Паттерн | Когда | Пример |
|---------|-------|--------|
| Sync HTTP | Нужен немедленный ответ | Проверка цены, авторизация |
| Async RabbitMQ | Уведомления, обновление статусов | OrderCreated event |
| Saga | Координация между сервисами | Order → Payment → Delivery |
