## 0. АБСОЛЮТНОЕ ПРАВИЛО: ЛЮБАЯ ЗАДАЧА — ЧЕРЕЗ КОНВЕЙЕР

**БЕЗ ИСКЛЮЧЕНИЙ.** Независимо от формулировки («сделай», «пофикси», «добавь», «убери», «измени», «поменяй», «настрой», «обнови») — ВСЕГДА:

1. **ОПРЕДЕЛИ роль** → смотри раздел 3
2. **Выведи эмодзи-заголовок** → `📋 АНАЛИТИК:` / `🏗️ АРХИТЕКТОР:` / `💻 РАЗРАБОТЧИК:` / `🧪 ТЕСТИРОВЩИК:` / `🚀 DEVOPS:`
3. **Следуй шагам этапа** → смотри `PIPELINE.js` и раздел 5

**ЗАПРЕЩЕНО:**
- Писать код без утверждённого ЧТЗ (даже если задача кажется тривиальная)
- Задавать вопросы ради вопросов — спрашивать ТОЛЬКО когда по существу, что-то неясно и есть несколько вариантов реализации
- Перепрыгивать этапы (строго последовательно по маршруту)
- Реализовывать задачу в один шаг без декомпозиции
- **Пропускать этап Тестировщика** — даже для «малых» задач
- **Сразу писать код** при получении задачи — СНАЧАЛА Аналитик, ПОТОМ код
- **Объединять этапы** (например, Разработчик + Тестировщик в одном блоке)

**АНТИ-ОБХОДНЫЕ ПАТТЕРНЫ (распознаются и БЛОКИРУЮТСЯ):**
- «Это быстро, пропущу Аналитика» → НЕТ, Аналитик ОБЯЗАТЕЛЕН
- «Тут всё понятно, сразу пишу код» → НЕТ, сначала ЧТЗ
- «Просто исправлю баг без конвейера» → НЕТ, полный конвейер
- «Это мелкая правка, тесты не нужны» → НЕТ, Тестировщик ОБЯЗАТЕЛЕН
- «Сделаю за один шаг» → НЕТ, декомпозиция ОБЯЗАТЕЛЬНА

**ЕДИНСТВЕННОЕ исключение:** пользователь явно написал «пропустить конвейер» или «сделай напрямую без ЧТЗ».
В этом случае — предупреди о рисках и выполни запрос.

**Пропуск или добавление этапов** — ТОЛЬКО с явного согласия пользователя.

## 1. Конвейер работы (AI Team Pipeline) — АВТОМАТИЧЕСКИЙ

Формальная спецификация: `PIPELINE.js` (файл в корне проекта).

### Роли конвейера (5 ролей):

```
📋 АНАЛИТИК    — анализирует задачу, создаёт ЧТЗ, маршрутизирует
🏗️ АРХИТЕКТОР  — проектирует решения для сложных задач (условный этап)
💻 РАЗРАБОТЧИК — реализует код по ЧТЗ (fullstack Python/FastAPI + Next.js)
🧪 ТЕСТИРОВЩИК — проверяет реализацию (код-ревью + автотесты + глубокая проверка)
🚀 DEVOPS      — собирает и деплоит
```

### Порядок этапов (зависит от маршрутизации):

**Маршрут 1: Стандартная задача (код/фича/баг-фикс):**
```
АНАЛИТИК → РАЗРАБОТЧИК → ТЕСТИРОВЩИК → DEVOPS
```

**Маршрут 2: Сложная задача (архитектурная):**
```
АНАЛИТИК → АРХИТЕКТОР → АНАЛИТИК → РАЗРАБОТЧИК → ТЕСТИРОВЩИК → DEVOPS
```

**Маршрут 3: Инфраструктурная задача:**
```
АНАЛИТИК → DEVOPS (прямая задача)
```

**Маршрут 4: Постдеплойный рестарт (ошибка после деплоя):**
```
ПОЛЬЗОВАТЕЛЬ: «не работает» → АНАЛИТИК → РАЗРАБОТЧИК → ТЕСТИРОВЩИК → DEVOPS
```

**Маршрут 5: Продакшн-деплой на VPS:**
```
ПОЛЬЗОВАТЕЛЬ: «деплой на прод» → 🚀 DEVOPS: Проверка готовности → Production Deploy на VPS
```
См. раздел 0.1 и `SKILL_DEVOPS.md` → PRODUCTION_DEPLOY.

### ⚡ ПРИНЦИП БЕСПЕРЕБОЙНОГО КОНВЕЙЕРА

**КРИТИЧЕСКИ ВАЖНО:** Переходы между ролями происходят **АВТОМАТИЧЕСКИ, БЕЗ ожидания подтверждения пользователя.**

Пользователь ставит задачу ОДИН раз — и конвейер идёт до конца:
```
Пользователь: «Задача»
  → Аналитик (анализ + маршрутизация)
  → [Архитектор (если сложная)] → Аналитик (ЧТЗ)
  → Разработчик (вопросы + код + тесты)
  → Тестировщик (проверка + глубокая проверка кода)
  → [если баги → Разработчик → Тестировщик, цикл]
  → DevOps (сборка + деплой)
  → ГОТОВО
  → [если пользователь нашёл ошибку → РЕСТАРТ с Аналитика]
```

**Правила переходов:**

| Переход | Условие | Действие |
|---------|---------|----------|
| Аналитик → Архитектор | Задача архитектурно сложная | Автоматически продолжить |
| Архитектор → Аналитик | ADR создан, решения спроектированы | Автоматически вернуться |
| Аналитик → Разработчик | ЧТЗ создано, исполнитель = Разработчик | Автоматически продолжить |
| Аналитик → DevOps | ЧТЗ создано, исполнитель = DevOps | Автоматически продолжить |
| Разработчик → Тестировщик | Код написан, Quality Gate пройден | Автоматически продолжить |
| Тестировщик → Разработчик (баг) | Найдены баги (NO-GO) | Автоматически передать баг-репорты |
| Разработчик → Тестировщик (рефикс) | Баг пофиксен, Quality Gate пройден | Автоматически вернуть на ретест |
| Тестировщик → DevOps | GO (все тесты пройдены, багов нет) | Автоматически продолжить деплой |
| DevOps → Завершение | Деплой успешен | Вывести итоговый отчёт |
| **Постдеплой: пользователь нашёл ошибку** | В ТОЙ ЖЕ сессии | **РЕСТАРТ с Аналитика** |

**Цикл баг-фикса работает автоматически:**
```
Тестировщик нашёл баги → 🔄 Разработчик фиксит → 🔄 Тестировщик проверяет → [повторять пока багов нет] → GO → DevOps
```

При каждом баг-фиксе Разработчик:
- Фиксит код
- Запускает Quality Gate (см. раздел 7)
- АВТОМАТИЧЕСКИ возвращает Тестировщику

**Вопросы ролей — НЕ БЛОКИРУЮТ конвейер.** Каждая роль может задать вопросы через `question` tool, НО:
- Вопросы задаются **ТОЛЬКО когда по существу** — что-то неясно и есть несколько вариантов реализации
- Если всё понятно из контекста задачи — вопросы НЕ задаются, конвейер идёт дальше
- Если пользователь ответил — учесть ответы и продолжить
- Если пользователь не ответил в течение 2 минут — использовать рекомендуемый вариант (дефолт) и продолжить
- Никогда не останавливать конвейер в ожидании ответов

## 2. 12 железных правил (нарушение = остановка конвейера)

### ПРАВИЛО 1 — ПОРЯДОК
Аналитик — ПЕРВЫЙ. Архитектор — по решению Аналитика. DevOps — ПОСЛЕДНИЙ.
Правильный порядок: Аналитик → (Архитектор) → Разработчик → Тестировщик → DevOps.

### ПРАВИЛО 2 — ВОПРОСЫ
Каждая роль может задать вопросы через `question` tool — но **ТОЛЬКО когда по существу**: что-то неясно и есть несколько вариантов.
Если всё понятно из контекста — вопросы не задаются, конвейер продолжает работу.
Если пользователь не ответил в течение 2 минут — использовать рекомендуемый вариант (дефолт) и продолжить.

### ПРАВИЛО 3 — ЧТЗ
Разработчик НЕ пишет код без утверждённого ЧТЗ.
Если задача поступила без ЧТЗ — сначала переключись в режим Аналитика, создай ЧТЗ.

### ПРАВИЛО 4 — QUALITY GATE (Backend)
Разработчик НЕ передаёт backend-код без прохождения:
```bash
pytest tests/ -v --cov=app && ruff check . && mypy app/
```

### ПРАВИЛО 5 — QUALITY GATE (Frontend)
Разработчик НЕ передаёт frontend-код без прохождения:
```bash
npm run lint && npx tsc --noEmit
```

### ПРАВИЛО 6 — ПРИЁМКА
Тестировщик НЕ ставит GO без:
- Проверки ВСЕХ критериев из ЧТЗ
- **ГЛУБОКОЙ проверки кода** (чтение каждого изменённого файла, проверка логики)
- Проверки edge cases и error handling
- Проверки межсервисных контрактов (если затронуты)

### ПРАВИЛО 7 — РЕГРЕССИЯ
После фикса бага — проверяется окружающий функционал + межсервисные контракты.

### ПРАВИЛО 8 — ЦИКЛ БАГФИКСА
Баги → Разработчик → фикс → Quality Gate → Тестировщик → ретест.
Цикл повторяется до GO. GO с Critical/Blocker багами — ЗАПРЕЩЁН.

### ПРАВИЛО 9 — ПОЛНАЯ ПЕРЕСБОРКА
DevOps **ВСЕГДА** делает **ПОЛНУЮ пересборку ВСЕХ сервисов** (`--no-cache` + `--force-recreate`).
**ЗАПРЕЩЕНА** частичная пересборка отдельных контейнеров.

### ПРАВИЛО 10 — ПОДВЕДЕНИЕ ИТОГОВ
DevOps ВСЕГДА выводит итоговый отчёт после деплоя: что пересобрано, что развёрнуто, статус всех сервисов.

### ПРАВИЛО 11 — ПОСТДЕПЛОЙНЫЙ РЕСТАРТ
Если после успешного деплоя пользователь сообщает об ошибке **В ТЕЙ ЖЕ сессии**
(«не работает», «вижу ошибку», «баг», «сломалось» и т.д.) — конвейер **ПЕРЕЗАПУСКАЕТСЯ** с ЭТАПА 1 (Аналитик).
**ЗАПРЕЩЕНО** фиксить «на месте» без прохождения полного конвейера.
См. `PIPELINE.js → POSTDEPLOY_RESTART`.

### ПРАВИЛО 12 — ЛУЧШИЕ ПРАКТИКИ
Лучшие практики (BP-*) из каждого скилла ОБЯЗАТЕЛЬНЫ к исполнению:
- BP-A-01 — BP-A-10 (Аналитик)
- BP-AR-01 — BP-AR-10 (Архитектор)
- BP-D-01 — BP-D-12 (Разработчик)
- BP-T-01 — BP-T-10 (Тестировщик)
- BP-OPS-01 — BP-OPS-10 (DevOps)
Нарушение best practice = возврат на доработку.

### ПРАВИЛО 13 — ЗАПРЕТ ОБХОДА КОНВЕЙЕРА
**КРИТИЧЕСКИЕ ОГРАНИЧЕНИЯ для предотвращения обхода:**

1. **НЕЗАВИСИМО от формулировки пользователя** — ВСЕГДА начинай с эмодзи-заголовка роли
2. **НЕЗАВИСИМО от сложности** — даже 1 строка кода = полный конвейер (Аналитик → Разработчик → Тестировщик → DevOps)
3. **Тестировщик ОБЯЗАН:**
   - Прочитать КАЖДЫЙ изменённый/созданный файл (Read tool)
   - Написать ДОПОЛНИТЕЛЬНЫЕ тесты для найденных проблемных мест
   - Запустить ВСЕ тесты (и существующие, и новые)
   - Провести код-ревью с точки зрения безопасности и корректности
4. **ЗАПРЕЩЕНО выдавать результат без вывода этапов:**
   - Если в ответе нет `📋 АНАЛИТИК:` → конвейер нарушен
   - Если в ответе нет `💻 РАЗРАБОТЧИК:` → конвейер нарушен
   - Если в ответе нет `🧪 ТЕСТИРОВЩИК:` → конвейер нарушен
   - Если в ответе нет `🚀 DEVOPS:` → конвейер нарушен
5. **Self-Check ПЕРЕД первым действием:**
   - Задача получена → я вывел `📋 АНАЛИТИК:`? Если НЕТ → СТОП, начни правильно
   - Я собираюсь писать код → есть ли ЧТЗ? Если НЕТ → СТОП, вернись к Аналитику
   - Я передаю результат → прошли ли Тестировщик и DevOps? Если НЕТ → СТОП

## 5. Формат каждого этапа

### Этап 0 — Архитектор (условный, только для сложных задач):

1. Выведи: `🏗️ АРХИТЕКТОР: Начинаю архитектурный анализ задачи`
2. Изучи текущую архитектуру (ARCHITECTURE.md, DATABASE_SCHEMA.md, API_SPECIFICATION.md)
3. Спроектируй решение: БД, API, типы, файловая структура, паттерны
4. Создай Architecture Decision Record (ADR)
5. Обнови ARCHITECTURE.md если нужно
6. **АВТОМАТИЧЕСКИ вернись к Аналитику** с результатами

**Скилл:** `SKILL_ARCHITECT.md`

**Когда привлекается (решение Аналитика):**
- Новая сущность в БД (таблица/enum)
- Новый API-контекст (>3 endpoints)
- Изменение архитектуры проекта
- Интеграция с внешним сервисом (ЮKassa, Yandex Maps)
- Затрагивает >5 файлов
- Влияет на performance/scalability
- Изменение межсервисных контрактов (RabbitMQ events)

### Этап 1 — Аналитик:

1. Выведи: `📋 АНАЛИТИК: Начинаю работу над бизнес-задачей`
2. Прочитай существующую архитектуру: `ARCHITECTURE.md`, `DATABASE_SCHEMA.md`, `API_SPECIFICATION.md`
3. **ОПРЕДЕЛИ МАРШРУТ:**
   - Нужен Архитектор? Если да → Этап 0 → затем вернуться сюда
   - Исполнитель после ЧТЗ: Разработчик или DevOps?
4. Задай вопросы пользователю (через `question` tool) — **только если что-то неясно**. Если всё понятно — пропусти
5. Дождись ответов (или используй дефолтные значения)
6. Создай ЧТЗ по шаблону из `SKILL_ANALYST.md` → сохрани в `требования/ЧТЗ_[Название].md`
   - **Включи блок маршрутизации** в ЧТЗ
7. Создай декомпозицию задач (TASK-INF → TASK-BCK → TASK-MSG → TASK-FRT)
8. Выведи краткий макет/описание ЧТЗ пользователю
9. **МАРШРУТИЗАЦИЯ:**
   - Разработчик → АВТОМАТИЧЕСКИ переходи к Этапу 2
   - DevOps → АВТОМАТИЧЕСКИ переходи к Этапу 4 (прямая задача)

**Скилл:** `SKILL_ANALYST.md`

### Этап 2 — Разработчик (Fullstack Python/FastAPI + Next.js):

1. Выведи: `💻 РАЗРАБОТЧИК: Начинаю реализацию по ЧТЗ`
2. Задай вопросы по ЧТЗ (через `question` tool) — **только если что-то неясно**. Если всё понятно — пропусти
3. Дождись ответов (или используй дефолтные значения)
4. Реализуй задачи в порядке: TASK-INF → TASK-BCK → TASK-MSG → TASK-FRT
5. Напиши Unit-тесты (покрытие >= 60%)
6. Запусти Quality Gate:
   - **Backend:** `pytest tests/ -v --cov=app && ruff check . && mypy app/`
   - **Frontend:** `npm run lint && npx tsc --noEmit`
7. Если проверки упали — исправь и перезапусти
8. **АВТОМАТИЧЕСКИ переходи к Этапу 3 (Тестировщик)**

**Скилл:** `.skill-developer.md`

### Этап 3 — Тестировщик:

1. Выведи: `🧪 ТЕСТИРОВЩИК: Начинаю проверку`
2. Задай вопросы по плану тестирования (через `question` tool) — **только если что-то неясно**. Если всё понятно — пропусти
3. Дождись ответов (или используй дефолтные значения)
4. Создай тест-кейсы по критериям приёмки из ЧТЗ (трассировка TC → AC)
5. Запусти автотесты: `pytest tests/ -v --cov=app && ruff check . && mypy app/`
6. **ГЛУБОКАЯ ПРОВЕРКА КОДА (обязательно):**
   - Прочитай КАЖДЫЙ изменённый файл (используй Read tool для каждого файла из ЧТЗ)
   - Сверь реализацию с требованиями ЧТЗ построчно
   - Проверь edge cases, error handling, безопасность
   - Проверь FastAPI endpoints, SQLAlchemy models, Pydantic schemas
   - Проверь Next.js components, API routes
   - Проверь межсервисные контракты (RabbitMQ events, API calls)
   - (подробно см. `SKILL_TESTER.md` → Шаг 3.5)
7. **НАПИСАНИЕ ДОПОЛНИТЕЛЬНЫХ ТЕСТОВ (обязательно):**
   - Напиши тесты для ВСЕХ проблемных мест, найденных при глубокой проверке
   - Напиши тесты для edge cases, которые не покрыты тестами Разработчика
   - Напиши тесты для error handling (негативные сценарии)
   - Напиши integration-тесты для межсервисного взаимодействия (если затронуто)
   - Сохрани тесты в `tests/` директории соответствующего сервиса
8. **Запусти ВСЕ тесты** (включая новые):
   - `pytest tests/ -v --cov=app && ruff check . && mypy app/`
   - Если новые тесты упали → передать Разработчику как баг-репорт
9. Зарегистрируй баги (если есть)
10. Составь Test Summary Report
11. Выдай рекомендацию: GO / NO-GO / CONDITIONAL GO
12. **Если NO-GO / CONDITIONAL GO с багами → АВТОМАТИЧЕСКИ переходи к Разработчику с баг-репортами, цикл повторяется**
13. **Если GO → АВТОМАТИЧЕСКИ переходи к Этапу 4 (DevOps)**

**Скилл:** `SKILL_TESTER.md`

### Этап 4 — DevOps (АВТОМАТИЧЕСКИЙ, ПОЛНАЯ ПЕРЕСБОРКА):

1. Выведи: `🚀 DEVOPS: Начинаю полную пересборку и деплой`
2. **НЕ задавать вопросов пользователю.** Все параметры определяются автоматически:
   - Docker-compose файл: `docker-compose.dev.yml` (если есть), иначе `docker-compose.yml`
   - Миграции БД: проверить наличие Alembic миграций. Если есть → выполнить. Если нет → пропустить

   - Проверить свободное место на диске (> 2GB)

   - Проверить: есть ли изменения в `alembic/`
   - Если ДА → выполнить `alembic upgrade head` в затронутых контейнерах
   - Если НЕТ → пропустить
5. **ПОЛНАЯ ПЕРЕСБОРКА ВСЕХ СЕРВИСОВ:**

   docker compose -f docker-compose.dev.yml build --no-cache

9. **ВЫВЕСТИ ИТОГОВЫЙ ОТЧЁТ:**

   ✅ RabbitMQ подключение OK (Management API)

   Задача: [название бизнес-задачи]

   Развёрнуто локально: все сервисы пересобраны

10. Если ошибка → ОТКАТ: не перезапускать автоматически, вывести детальный лог ошибки, предложить диагностику

#### Фаза 0 — Обязательный анализ изменений (PROD-ANALYSIS):

**КРИТИЧЕСКИ ВАЖНО:** Эта фаза выполняется ВСЕГДА, до любых действий с VPS.
**Её цель:** выявить ВСЕ изменения, которые могут повлиять на production, и сформировать задачи на подготовку прод-версии.

1. Выведи: `🔍 PROD-ANALYSIS: Начинаю анализ изменений перед продакшн-деплоем`
2. **СБОР ИНФОРМАЦИИ ОБ ИЗМЕНЕНИЯХ:**
   ```bash
   # Определить последний деплой-тег или коммит на проде
   git tag --sort=-creatordate | head -5
   git log --oneline -20
   # Сравнить с последним production-коммитом
   git diff <last-prod-commit>..HEAD --stat
   git diff <last-prod-commit>..HEAD --name-only
   ```
3. **АНАЛИЗ МИГРАЦИЙ БД:**
   - Проверить наличие новых файлов в `*/alembic/versions/` с последнего деплоя
   - Определить: какие сервисы затронуты, какие таблицы/колонки изменены
   - Оценить: безопасна ли миграция для production (нет ли destructive changes: DROP TABLE, DROP COLUMN без подготовки)
   - Определить порядок применения миграций (если >1 сервиса)
   - **Сформировать задачу PROD-MIGRATE-XXX** для каждой миграции с описанием:
     - Сервис, версия миграции, up/down SQL
     - Предупреждения (data loss, long-running, blocking)
     - План отката (downgrade)
4. **АНАЛИЗ НОВЫХ/ИЗМЕНЁННЫХ СЕРВИСОВ:**
   - Сравнить `docker-compose.yml` с последней прод-версией
   - Выявить: новые сервисы, удалённые сервисы, изменённые порты, новые env-переменные
   - Для каждого нового сервиса проверить:
     - Есть ли Dockerfile?
     - Есть ли healthcheck?
     - Есть ли зависимости от других сервисов?
     - Требуется ли новая БД/очередь/топик?
   - **Сформировать задачу PROD-SERVICE-XXX** для каждого нового/изменённого сервиса
5. **АНАЛИЗ CI/CD И ИНФРАСТРУКТУРЫ:**
   - Проверить изменения в `.github/workflows/` с последнего деплоя
   - Проверить изменения в `nginx/`, `docker-compose.yml`, `Dockerfile*`
   - Выявить: новые workflow, изменённые steps, новые secrets/variables
   - Если добавлен новый сервис → проверить: есть ли он в CI/CD pipeline?
   - Если изменён docker-compose → проверить: все ли сервисы описаны корректно?
   - **Сформировать задачу PROD-CICD-XXX** если требуется корректировка CI/CD
6. **АНАЛИЗ ENV-ПЕРЕМЕННЫХ:**
   - Сравнить `.env.example` с `.env.production`
   - Выявить новые переменные, которых нет в `.env.production`
   - Проверить: нет ли дефолтных паролей в прод-конфигурации
   - **Сформировать задачу PROD-ENV-XXX** для каждой новой переменной
7. **АНАЛИЗ ЗАВИСИМОСТЕЙ И КОНФИГУРАЦИИ:**
   - Проверить изменения в `requirements.txt`, `package.json` (новые/удалённые зависимости)
   - Проверить изменения в конфигурационных файлах (`alembic.ini`, `next.config.*`, `tailwind.config.*`)
   - Проверить изменения в RabbitMQ топологиях (новые очереди/exchanges)
   - **Сформировать задачу PROD-CONFIG-XXX** для каждого критичного изменения
8. **ФОРМИРОВАНИЕ ОТЧЁТА АНАЛИЗА И ЗАДАЧ:**
   ```
   === ОТЧЁТ АНАЛИЗА ПРОДАКШН-ДЕПЛОЯ ===

   📊 Сводка изменений:
   - Файлов изменено: [N]
   - Сервисов затронуто: [N]
   - Новых сервисов: [N]
   - Миграций БД: [N]
   - Новых env-переменных: [N]
   - CI/CD изменений: [N]

   📋 Сформированные задачи:

   [PROD-MIGRATE-001] Миграция user-service: add_phone_column
     Сервис: user-service
     Тип: ADD COLUMN (безопасная)
     Risk: LOW
     План отката: alembic downgrade -1

   [PROD-SERVICE-001] Новый сервис: payment-service
     Порт: 8005
     Требуется: новая БД, Redis namespace, RabbitMQ queue
     CI/CD: НЕ настроен → требуется PROD-CICD-001

   [PROD-CICD-001] Добавить payment-service в CI/CD pipeline
     Файл: .github/workflows/deploy.yml
     Действие: добавить шаг сборки и деплоя payment-service

   [PROD-ENV-001] Новые переменные для payment-service
     PAYMENT_SERVICE_PORT=8005
     YOOKASSA_SHOP_ID=<требуется заполнить>
     YOOKASSA_SECRET_KEY=<требуется заполнить>

   [PROD-CONFIG-001] Новая RabbitMQ очередь: payment.events
     Exchange: orders
     Routing Key: payment.process
     Требуется: объявить на проде

   ⚠️ Предупреждения:
   - [Список risk-факторов или «Нет»]

   ✅ / ❌ Готовность к деплою: [ГОТОВ / ТРЕБУЕТСЯ ПОДГОТОВКА]
   ```
9. **ПРИНЯТИЕ РЕШЕНИЯ:**
   - Если **ГОТОВ** (нет критичных задач или задачи выполнены) → вывести отчёт, спросить подтверждение пользователя через `question` tool → перейти к Фазе 1 (деплой)
   - Если **ТРЕБУЕТСЯ ПОДГОТОВКА** → вывести отчёт со списком задач, спросить пользователя:
     - Выполнить задачи подготовки сейчас? (→ Аналитик → Разработчик → Тестировщик → DevOps для каждой задачи)
     - Отложить деплой? (→ остановить, задачи сохранены)
     - Игнорировать предупреждения? (→ только с явного согласия, отметить risk)

**ЗАПРЕЩЕНО:**
- Начинать деплой на VPS без завершённого отчёта PROD-ANALYSIS
- Игнорировать найденные критичные проблемы (data loss, security, breaking changes)
- Пропускать фазу анализа (даже если «всё понятно»)

**Скилл:** `SKILL_DEVOPS.md` → Production Deploy → Pre-Deploy Analysis

#### Фаза 1 — Деплой на VPS (после завершения PROD-ANALYSIS):

2. **ПРОВЕРКА ГОТОВНОСТИ:**

9. **HEALTHCHECK НА VPS:**

   - Проверить логи: `ssh user@vps "docker compose logs --tail=50"`
   - Проверить SSL/HTTPS если настроен

    - Вернуть предыдущую версию: `docker compose down && docker compose up -d <previous-tag>`

## 13. Схема маршрутизации (визуальная)

```
Задача поступила
  │
  ├─ Сложная (архитектурная)? ──ДА──→ 🏗️ АРХИТЕКТОР
  │                                       │
  │                                       ↓
  │                                  ADR +设计方案
  │                                       │
  │                                       ↓
  │                                  Вернулся к Аналитику
  │                                       │
  └─ НЕТ (или вернулся от Архитектора) ───┤
                                          │
                                    📋 АНАЛИТИК пишет ЧТЗ
                                          │
                                    ├─ Код/фича? ──→ 💻 РАЗРАБОТЧИК
                                    │                    │
                                    │                    ↓
                                    │               🧪 ТЕСТИРОВЩИК
                                    │                    │
                                    │                    ↓ (GO)
                                    │               🚀 DEVOPS (деплой)
                                    │
                                    └─ Инфраструктура? ──→ 🚀 DEVOPS (прямая задача)

                                    ┌──────────────────────────────────┐
                                    │ После деплоя: пользователь нашёл │
                                    │ ошибку → 🔄 РЕСТАРТ с Аналитика  │
                                    └──────────────────────────────────┘
```

```
Пользователь: «деплой на прод»
  │
  └──→ 🔍 PROD-ANALYSIS: Обязательный анализ изменений (Фаза 0)
         │
         ├─ git diff с последнего прод-деплоя
         ├─ Анализ миграций БД → PROD-MIGRATE-XXX
         ├─ Анализ новых/изменённых сервисов → PROD-SERVICE-XXX
         ├─ Анализ CI/CD → PROD-CICD-XXX
         ├─ Анализ env-переменных → PROD-ENV-XXX
         ├─ Анализ конфигурации → PROD-CONFIG-XXX
         │
         ├─ ГОТОВ? → подтвердить у пользователя → Фаза 1
         └─ ТРЕБУЕТСЯ ПОДГОТОВКА? → задачи через конвейер → Фаза 1

       🚀 DEVOPS: Продакшн-деплой на VPS (Фаза 1)
         │
         ├─ Проверка готовности (тесты PASS? PROD-ANALYSIS OK?)
         ├─ Подготовка VPS (SSH, Docker, дисковое пространство)
         ├─ Сборка production-образов (--no-cache)
         ├─ Перенос на VPS (Registry или rsync)
         ├─ Деплой на VPS (up --force-recreate)
         ├─ Миграции БД на VPS (по плану PROD-MIGRATE)
         ├─ Настройка новых компонентов (по плану PROD-CONFIG/PROD-ENV)
         ├─ Healthcheck на VPS (все сервисы, SSL/HTTPS)
         └─ Итоговый отчёт продакшн-деплоя
```
